## Architectural correction summary (the cycle's core arc)

The cycle began with aweb 1.18.3 shipping aalk + aalm under "wholesale
closure" framing for cross-team-cert messaging. Empirical attestation came
back RED on Amy's actual workspace shape — exposing a sequence of
architectural concerns:

1. **Mode 1** (population-level): server's elif-to_stable_id binding-check
   compared local agent_id rather than canonical did_aw. Fixed in d4fb982
   (1.18.4 / v0.5.8.1 hotfix).
2. **Mode 2 surface bugs**: signEnvelope fail-open on resolver errors;
   awid LIST endpoint correctness; sub-binding asymmetries. Fixed across
   1.18.4-1.18.5.
3. **The deepest correction — trust model drift**: AWID's address-read
   visibility filter was using `EXISTS (SELECT 1 FROM team_certificates ...)`
   to authorize private reads. This made AWID the membership oracle and
   centralized trust state at the registry — exactly what the
   decentralized-trust value proposition was designed to avoid. The fix
   (1.18.6 + 0.5.2): cert-presentation + signature verification +
   revocation check; AWID stores team **public keys** for verification but
   never uses cert row existence as the authorization predicate.

**Value proposition preserved**: a user who controls a DNS-verified
namespace can register their team's public key + manage their team's
revocations at AWID. Anything beyond that (issuing certs, distributing
them, member listing) the team handles itself. AWID is bounded to public
keys + revocation events. The decentralization holds because AWID doesn't
gate adding/removing members — only verifying signatures against stored
team keys.
